# EconTopic 文件结构设计

## 核心问题

> 这个项目涉及 Python 爬虫、TypeScript 前后端、LLM Prompt、Claude Code Skills/Agents、数据库、文档——这么多不同类型的文件，该如何组织？

---

## 一、架构决策：混合型

### 为什么 Python？

gov.cn 可能有反爬，Python 生态（BeautifulSoup、Playwright、Scrapy）在爬虫上远比 Node.js 成熟。即使当前 gov.cn 有 JSON API，未来如果加了 Cloudflare 验证码，Python 方案更多。

### 为什么 TypeScript/Next.js？

Vercel 原生支持 Next.js。API Route 可以直接当后端，Cron Job 内置支持，一套代码前后端全搞定。对个人项目来说运维复杂度最低。

### 结论

| 层 | 用什么 | 放哪里 |
|---|---|---|
| 爬虫（开发/调试） | Python CLI 工具 | `scraper/` |
| 爬虫（生产管道） | TypeScript `fetch()` | `lib/pipeline.ts` |
| 经济筛选 + LLM 分析 | TypeScript（Vercel 上跑） | `lib/` |
| LLM Prompt 模板 | Markdown（跨语言共用） | `prompts/` |
| 数据库 | Prisma ORM | `prisma/` |
| Web 前端 | Next.js 16 App Router | `app/` + `components/` |
| Claude Code 工具 | Skills + Agents | `.claude/skills/` `.claude/agents/` |
| 中文规划文档 | Markdown | `提示词/` |

## 三、每个文件夹的职责边界

### `.claude/` — Claude Code 编排层
> 放什么：Skills（斜杠命令）和 Agents（子任务执行者）
> 不放什么：业务代码、配置、数据
> 谁在用：你在 Claude Code 里通过 `/命令名` 交互

### `app/` — Next.js 路由层
> 放什么：页面文件、API 路由、全局样式
> 不放什么：组件、工具函数、类型定义
> 谁在用：用户浏览器访问

### `components/` — UI 组件层
> 放什么：React 组件，`ui/` 下是 shadcn 自动生成的
> 不放什么：服务端代码、数据库查询、API 调用
> 谁在用：被 `app/` 中的页面引用

### `lib/` — 服务端逻辑层
> 放什么：数据库操作、API 调用、管道编排、工具函数
> 不放什么：UI 组件、页面
> 谁在用：被 `app/api/` 和 `app/page.tsx`（服务端组件）引用

### `prisma/` — 数据库定义层
> 放什么：`schema.prisma`（数据模型）和 `seed.ts`（测试数据）
> 不放什么：数据库连接逻辑（在 `lib/db.ts`）
> 谁在用：Prisma CLI 生成迁移 + `lib/db.ts` 引用

### `scraper/` — Python 开发工具
> 放什么：标准 Python 包。`cli.py` 提供命令行 `econtopic` 命令
> 不放什么：TypeScript 代码、前端组件
> 谁在用：你在终端用 `python -m econtopic_scraper.cli` 本地测试

### `prompts/` — LLM 提示词仓库
> 放什么：Markdown 格式的 System Prompt 模板
> 不放什么：代码、配置
> 谁在用：被 `lib/anthropic.ts` 和 `scraper/analyzer.py` 共同读取

### `types/` — TypeScript 类型
> 放什么：共享类型定义（`News`, `Analysis`, `Category` 等）
> 不放什么：实现代码
> 谁在用：被 `lib/`、`app/`、`components/` 全部引用

### `提示词/` — 中文规划文档
> 放什么：系统架构、方案对比、设计决策的中文文档
> 不放什么：代码、配置
> 谁在用：你回顾项目设计时阅读

## 四、Python 和 TypeScript 如何共存

```
                    ┌──────────────────────┐
                    │   Vercel Postgres     │  ← 唯一数据源
                    │   (Neon 云数据库)      │
                    └──────┬───────────────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         Prisma        psycopg2      Prisma
         (lib/db.ts)   (scraper/db)   (lib/db.ts)
              │            │            │
     ┌────────┴───┐  ┌───┴────────┐    │
     │ Next.js    │  │ Python CLI │    │
     │ Vercel 上   │  │ Mac 本地    │    │
     └────────────┘  └────────────┘    │
```

**关键点**：
- `prisma/schema.prisma` 是数据库结构的唯一定义
- Python 端的 `scraper/src/models.py` 定义等价的 Pydantic 模型，与 Prisma schema 保持同步
- 两边读写同一数据库，通过数据库交换数据
- Prompt 模板（`prompts/` 下）是 Markdown 文本，Python 和 TypeScript 各读各的

## 五、为什么不把所有东西放进 src/

常见做法是 `src/` 一锅炖，但本项目不适用：

| 原因 | 说明 |
|---|---|
| Python 和 Node 是两个世界 | `src/` 对 Python 没有意义，Python 用包名目录 |
| Next.js 有明确约定 | `app/`、`components/`、`lib/` 是 Next.js 约定俗成 |
| Skills 必须放 `.claude/` | Claude Code 硬性要求 |
| Prompt 要跨语言共用 | 顶层 `prompts/` 最直观，不会被埋没 |
| 中文文档是设计资产 | 你和 Claude 都要频繁查看，放 `提示词/` 醒目 |

## 六、实施顺序建议

```
第 1 步：创建所有空目录
  └── mkdir -p app/ components/ui/ lib/ prisma/ scraper/src/ prompts/ types/ public/
  └── mkdir -p .claude/skills/{economic-filter,paper-topic-analyzer,site-publisher,daily-pipeline}
  └── mkdir -p .claude/agents/

第 2 步：prisma/schema.prisma  ← 数据库模型最先建，因为是所有代码的契约

第 3 步：prompts/  ← LLM 提示词，Python 和 TS 都依赖

第 4 步：types/index.ts  ← 类型定义，其他 TS 文件都引用

第 5 步：lib/  ← 服务端逻辑（依赖上面 2-4 步）

第 6 步：app/api/  ← API 路由（依赖 lib/）

第 7 步：app/ + components/  ← 前端页面（依赖 API + components）

第 8 步：.claude/skills/ + .claude/agents/  ← Claude Code 工具（最后造，因为依赖前面稳定）

第 9 步：scraper/  ← Python CLI（与上面并行开发也可以）

第 10 步：vercel.json + .env.example + 配置文件
```
