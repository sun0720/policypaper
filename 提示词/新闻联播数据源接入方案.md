# 新闻联播数据源接入方案

## 目标

在现有 PolicyPaper 系统（中国政府网 → 经济论文选题）的基础上，新增**新闻联播**作为第二个新闻数据源：
- 从新闻联播网站抓取每日新闻
- 筛选与经济相关的新闻
- 基于经济新闻生成论文选题
- 发布到网站展示

---

## 一、整体策略：复用现有管道，新增抓取层

现有 `economic-filter`、`paper-topic-analyzer` 和 `site-publisher` 三个模块都是**与新闻源无关的**——它们只消费 JSON 格式的新闻数据，不关心新闻来自哪个网站。因此 75% 的工作已完成。

需要做的核心工作：
1. **新建 `cctv-scraper` Skill**（对标 `gov-scraper`）
2. **调整数据流和网站展示**以支持多源

---

## 二、新建 `cctv-scraper` Skill

完全对标 `gov-scraper` 的结构，位于 `.claude/skills/cctv-scraper/`：

```
.claude/skills/cctv-scraper/
├── SKILL.md                         ← Skill 定义
├── scripts/scrape_cctv.py           ← 核心抓取脚本
└── references/page-structure.md     ← 新闻联播网站页面结构文档
```

### 与 gov-scraper 的关键差异

- 新闻联播文本通常在 `tv.cctv.com` 或相关子域名，需先调研具体 URL 模式和 API 端点
- 输出 JSON 格式与 `gov-scraper` 保持一致，增加 `source: "cctv"` 字段
- 如果新闻联播文本较口语化（播音稿风格），在 `economic-filter` 分类时需要额外注意语境转换

### 输出 JSON 格式

```json
[
  {
    "title": "一季度中国经济稳中向好...",
    "url": "https://tv.cctv.com/...",
    "date": "2026-06-10",
    "content": "新闻联播报道，一季度国内生产总值...",
    "source": "cctv"
  }
]
```

## 三、数据目录结构调整

从单源变为双源，数据目录体现来源：

```
data/
├── raw/
│   ├── gov/                        ← 中国政府网（原有）
│   │   ├── 2026-06-10.json
│   │   └── ...
│   └── cctv/                       ← 新闻联播（新增）
│       ├── 2026-06-10.json
│       └── ...
├── filtered/
│   ├── gov/
│   │   └── 2026-06-10.json
│   └── cctv/
│       └── 2026-06-10.json
├── exports/
│   ├── 2026-06-10.md               ← 合并后的最终输出（不变）
│   └── ...
└── .cache/
    └── parsed.json
```

**核心原则**：raw 和 filtered 层分源存储，exports 层合并输出。网站展示时用户关注的是选题内容而非来源。

## 六、Pipeline 调整（核心改动）

### 新的双源流水线架构

```
                    ┌─ 🔍 Scrape-gov ────→ data/raw/gov/{date}.json
                    │
  🚀 Pipeline 触发 ──┤
                    │
                    └─ 🔍 Scrape-cctv ───→ data/raw/cctv/{date}.json
                            │
                    ┌───────┘
                    ▼
              🏷️ Filter-gov ──→ data/filtered/gov/{date}.json
              🏷️ Filter-cctv ─→ data/filtered/cctv/{date}.json
                    │
                    ▼
              🧩 合并两个 filtered JSON（按标题相似度去重）
                    │
                    ▼
              🧠 Analyze（并行逐条生成选题，每条新闻 1 个 agent）
                    │
                    ▼
              🧩 Combine → data/exports/{date}.md
                    │
                    ▼
              🚀 Publish → 网站部署
```

### 关键改动点

1. **Phase 1** 拆为两个并行的 agent（Scrape-gov + Scrape-cctv），各自独立抓取
2. **Phase 2** 同样并行过滤
3. 新增**合并步骤**：将两个 filtered JSON 合并为一个统一的输入文件
4. **去重逻辑**：两个源的新闻可能有重复（同一天同一政策两边都报道），合并时按标题相似度 >80% 视为重复，保留来源更权威的那条
5. Phase 3-5 不变，因为下游模块是源无关的

## 七、新增独立 Pipeline Skill（可选）

可以再建一个 `cctv-pipeline` Skill，专门跑「仅新闻联播」的流程，供独立调试和补跑使用：

```
cctv-scraper → economic-filter → paper-topic-analyzer → site-publisher
```

### 三种流水线对比

| Pipeline | 覆盖源 | 用途 |
|---|---|---|
| `policy-pipeline` | gov.cn + 新闻联播 | **日常全量运行** |
| `gov-pipeline`（现有） | 仅 gov.cn | 独立调试政府网 |
| `cctv-pipeline`（新增） | 仅新闻联播 | 独立调试新闻联播 |

## 九、关键风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| 新闻联播网站反爬或需登录 | 无法抓取 | 先用浏览器 DevTools 分析网络请求，找 JSON API；必要时使用 Playwright 无头浏览器 |
| 新闻联播内容偏政治/外交，经济新闻占比低 | 每天仅 0-2 条经济新闻 | 预期合并 gov.cn 后总量 5-10 条，够用；若太少可考虑再接入其他源 |
| 两个源新闻重复（同一政策两边都报道） | 选题重复 | 合并阶段按标题相似度去重（>80% 视为重复，保留来源更权威的那条） |
| 新闻联播文本口语化（播音稿风格） | 经济分类准确率下降 | 在 `economic-filter` 的 Prompt 中增加「播音稿转政务语境」的说明 |
| API 调用成本增加 | 月度费用上升 | 双源并行 → 新闻量翻倍 → API 调用翻倍；预估从 $30-60/月 升至 $50-100/月 |
