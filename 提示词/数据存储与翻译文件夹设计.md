# 数据存储与翻译文件夹设计

## 核心问题

> 爬下来的原始数据、清洗后的新闻、翻译内容、分析结果导出——这些文件放哪里？怎么组织？

---

## 一、三种数据，三种存储方式

| 数据类型 | 存储位置 | 格式 | 用途 |
|---|---|---|---|
| 临时文件（原始抓取、中间结果） | `data/raw/` `data/processed/` | JSON / CSV | 本地调试、验证爬虫 |
| 持久化数据（新闻、分析、标签） | Vercel Postgres（通过 Prisma） | 数据库表 | 网站展示、生产环境 |
| 翻译内容 | `data/translations/` 或数据库多语言字段 | JSON / DB | 双语展示、学术引用 |

**原则**：临时的放文件，持久的放数据库。

## 三、为什么不在 scraper/ 下放数据？

```
❌ scraper/data/raw/          ← 数据是项目资产，不只属于爬虫
❌ scraper/data/translations/ ← 翻译可能来自 TypeScript 端，不限于 Python
❌ scraper/data/exports/      ← 导出结果面向用户，与工具无关

✅ data/raw/                  ← 顶层 data/，清晰表明"这是项目数据"
✅ data/processed/
✅ data/translations/
✅ data/exports/
```

**对比**：

| | 放 scraper/data/ | 放顶层 data/ |
|---|---|---|
| Python 脚本引用 | `Path("data/raw")` 不够直观 | `Path("../../data/raw")` 路径长 |
| TypeScript 引用 | 无法访问（跨语言） | `fs.readFileSync("data/...")` 清晰 |
| 新增 Go/Rust 工具 | 又要新建目录 | 统一读写 data/ |
| 备份时 | 要在工具目录里翻找 | 直接打包 data/ 即可 |

**解决路径长的问题**：在 Python 包的 `cli.py` 里定义全局路径常量：

```python
# scraper/src/econtopic_scraper/cli.py
from pathlib import Path

PROJECT_ROOT = Path(__file__).parent.parent.parent.parent
DATA_DIR = PROJECT_ROOT / "data"
RAW_DIR = DATA_DIR / "raw"
PROCESSED_DIR = DATA_DIR / "processed"
TRANSLATIONS_DIR = DATA_DIR / "translations"
EXPORTS_DIR = DATA_DIR / "exports"
```

## 四、翻译功能的设计选项

### 选项 A：翻译作为独立步骤（推荐起步）

```
爬虫抓取 → 清洗 → 经济筛选 → 论文分析 → 翻译 → 发布
                                          ↑
                                    新增这一步
```

**翻译什么**：
- 新闻标题 + 摘要（中→英）
- 论文话题分析（中→英）
- 可选：关键词、分类标签

**怎么翻**：Claude API 翻译 Prompt（比 Google 翻译更懂学术语境）

```
翻译 Prompt 示意：
"你是经济学学术翻译专家。将以下中文经济新闻和论文分析翻译成英文，
 保持学术准确性和经济学专业术语的正确表达。
 保留专有名词（如机构名、政策名）的拼音+括号英文释义。"
```

### 选项 B：分析时直接双语输出

在 `paper-topic-generator` 的 Prompt 中要求同时输出中英文：

```
"请对每条论文方向同时提供中文版和英文版标题、研究问题和摘要。"
```

**对比**：

| | A：独立翻译步骤 | B：双语同时输出 |
|---|---|---|
| 质量 | 专注翻译，术语更准确 | 一次生成，可能互相干扰 |
| Token 成本 | 额外一次 API 调用 | 输出翻倍，但只调用一次 |
| 灵活性 | 可选择性翻译 | 每次都双倍输出 |
| 适合场景 | 选译高价值内容 | 全量覆盖 |

**建议**：先用 B 做双语标题和摘要（成本低），关键分析用 A 做精翻。

## 五、双语数据的数据库设计

如果要在网站上做双语展示，Prisma Schema 需要调整：

```prisma
model News {
  id            String   @id
  titleZh       String                       // 中文标题
  titleEn       String?                      // 英文标题（可选，翻译后填入）
  summaryZh     String                       // 中文摘要
  summaryEn     String?                      // 英文摘要
  url           String
  publishedAt   DateTime
  // ...
}

model PaperTopic {
  id            String   @id
  newsId        String
  titleZh       String                       // 中文论文标题
  titleEn       String?                      // 英文论文标题
  questionZh    String                       // 中文研究问题
  questionEn    String?                      // 英文研究问题
  frameworkZh   String
  frameworkEn   String?
  // ...
}
```

或者用更简洁的单表翻译关联：

```prisma
model Translation {
  id        String @id
  tableName String // "News" | "PaperTopic"
  recordId  String
  field     String // "title" | "summary" | "question"
  lang      String // "zh" | "en"
  content   String
}
```

**推荐**：直接用第一种（字段级双语），简单直观，查询不需要 JOIN。

## 六、完整的数据流（含翻译）

```
本地开发：
  python scraper/scripts/daily_run.py --date 2026-06-05
    → 抓取 → data/raw/2026-06-05/gov_news_list.json
    → 清洗 → data/processed/2026-06-05/news_cleaned.json
    → 筛选 → data/processed/2026-06-05/economic_filtered.json
    → 分析 → (写数据库 + exports 导出)
    → 翻译 → data/translations/2026-06-05/en/
    → 导出 → data/exports/2026-06-05/daily_report.md

生产环境（Vercel Cron）：
  GET /api/cron/daily
    → 抓取（内存） → 筛选（内存） → 分析 → 翻译 → 写数据库
    → 不产生文件，全部在数据库内流转

长期归档：
  每月执行：打包 data/raw/ + data/processed/ + data/translations/
    → data/archive/2026-06/monthly_archive.zip
```

